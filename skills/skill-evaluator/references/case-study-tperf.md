# 案例：TPerf 性能 AI 分析 Agent 测评实践

> 来源：腾讯 TEG 云架构平台部 网关测试团队
>
> 完整原文：https://mp.weixin.qq.com/s/PUbGqheJhFMmb6hGj1ZtOw

## Agent 概况

- **类型**：功能工具
- **平台**：Knot 智能体平台
- **能力**：通过 MCP 调用 TPerf-API 获取性能数据，结合知识库进行深度分析，经决策树判断资源瓶颈和压测有效性，输出结构化性能分析报告
- **调用链路**：用户触发 → Agent 拉取压测数据 → 检索知识库 → 分析瓶颈 → 输出报告

## 测评设计

### 评委组合

**两层组合**：确定性评分（脚本） + Rubric 评分（LLM）

| 维度 | 评分方法 | 最大扣分 |
|---|---|---|
| 操作步骤 | LCS 算法对齐基线与 Trial 步骤，不一致/多余/缺少/顺序错误各 -2 | -10 |
| 效率 | 耗时以 300s 为基准每超标 10% -1，Token 以基线为基准每超标 10% -1 | -10 |
| 关键判定 | LLM 对比压测有效性和性能结果 | **-80**（不一致直接扣） |
| 章节内容 | LLM 逐项对比 10 个子项，每项差异 -5 | -80（上限） |
| 稳定性 | N 次并发执行，全部达标取平均分，任一不达标 0 分 | — |

### 稳定性策略

TPerf 虽属辅助分析类（理论容忍 ≤10%），但性能分析结论直接影响上线决策，选择了更严格的**关键决策类标准**——任一 Trial 不达标即判定不通过。

### 用例集

按性能瓶颈场景分组，9 大类 30+ 用例，覆盖真实压测记录：

- CPU 瓶颈（6 个）
- 网卡瓶颈（5 个）
- 磁盘瓶颈（4 个）
- 磁盘 IO（2 个）
- 内存（1 个）
- TCP 队列（3 个）
- Nginx（1 个）
- 配置问题（1 个）
- 正常场景（5 个）

每个用例 × 7 个模型生成独立参数化测试，全量测评耗时约 3-5 小时。

## 基线管理

基线以 Knot 会话标识（session_id + message_id）保存，评测时通过 API 动态获取完整数据：

```
基线内容:
├── 分析报告（完整 Markdown）
├── 操作步骤（工具名 + 简化参数 + 调用顺序）
├── 执行耗时（秒）
└── Token 消耗（输入/输出/总计）
```

基线更新触发：Agent 逻辑变更、外部依赖变更、模型升级、新增用例/Bad Case、新增分析能力。

## 工程实现

### 项目分层

| 层 | 职责 |
|---|---|
| API 封装层 | Knot API + TPerf API |
| 配置层 | 全局参数、凭证、模型定价 |
| 用例定义层 | 按场景分组的 YAML 文件 |
| 采集层 | 基线数据采集 |
| 评分层 | 三个独立评分器（步骤/效率/报告） |
| 执行层 | 编排流程、并发控制 |
| 解析层 | AG-UI 事件流 → 结构化步骤 |
| 结果层 | 落盘、归档、HTML 报告 |

### 执行模式

| 模式 | 说明 |
|---|---|
| 快速分析 | 每用例 1 轮，生成可视化报告 |
| 全量测评 | 每用例 5 轮 × 7 模型，评估稳定性 |
| 按模型筛选 | 只运行指定模型的用例 |
| 重新评分 | 对已有结果重新评分（无需重新执行） |

### 流水线

- **基线确认流水线**：日常开发调试，确定基线，左右分栏 HTML 报告
- **测评流水线**：pytest + TCase 平台，并发执行+并行评分，自动重试+超时控制

### 报告内容

- 顶部：通过率、平均分、模型版本、Token/费用汇总
- 分组展示：按场景分组，每用例显示得分和 Trial 分布直方图
- 单用例详情：Trial 三维度评分明细 + 基线对比
- 稳定性评分：达标情况和最终分数
- 对话链接：每个 Trial 的 Knot 平台跳转
- 模型对比：多模型横向对比表

## 关键经验

1. **用例用真实数据**：TPerf 压测记录一旦生成不可变，确保可复现
2. **并发是关键**：Trial 并行 + 评分并行 + 三个评分器并行
3. **过程归档**：每次执行所有中间数据归档至流水线产物，确保可追溯
4. **重新评分无需重跑**：评分逻辑调整后可直接对已有结果重新评分
5. **基线随代码版本化**：基线配置随 Git 追溯，执行结果流水线归档
