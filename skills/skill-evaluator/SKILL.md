# Skill Evaluator

为 Agent Skill 建立可重复、可量化、可持续演进的运行时测评闭环。

核心公式：

> **Eval = Agent 输入 → 执行 → 捕获 Trace + 产物 → 一组检查规则 → 可对比的分数**

## 核心理念

测评的目标不是追求完美覆盖，而是**每次变更都能快速跑出一个可比较的分数，用数据代替直觉，用全量代替抽查**。

本技能覆盖从测评框架设计到工程落地的全流程。根据你的需求，可以完整执行全部步骤，也可以只执行其中某几个。

## 工作流

按以下顺序推进，每步可独立执行：

### 1. 了解被测对象

先搞清楚要测什么：

- **Skill 类型**：知识库问答 / 代码编写 / 功能工具 / 问题定位？
- **核心能力**：这个 Skill 做什么？输入输出是什么？
- **关键风险**：哪里最容易出错？（工具调用？幻觉？效率？稳定性？）
- **使用场景**：用户怎么触发它？有哪些典型 prompt？

不同类型的 Skill 测评侧重不同（详见 `references/evaluation-dimensions.md`）：

| Skill 类型 | 测评侧重 | 典型检查项 |
|---|---|---|
| 知识库问答 | 准确性、幻觉检测、引用溯源 | 回答是否基于知识库而非编造 |
| 代码编写 | 产物正确性、可运行性 | 代码是否编译通过、测试是否跑过 |
| 功能工具 | 过程合规性、工具调用正确性 | 是否按预期调用正确工具、参数是否正确 |
| 问题定位 | 推理链路、根因准确性 | 推理是否逻辑清晰、是否定位到真实根因 |

### 2. 选择评委组合

没有银弹评分器，必须三类组合使用。选择优先级：**确定性 > Rubric > 人工**——能用代码判断的绝不用模型。

#### 三类评委

| 评委 | 谁来评 | 规则在哪 | 成本 | 稳定性 | 适用场景 |
|---|---|---|---|---|---|
| **确定性评分器** | 脚本 | 代码里 | 毫秒级/免费 | 100% | 文件存在、构建通过、工具调用检查、Token/耗时阈值 |
| **Rubric 评分器** | 大模型(固定版本) | Prompt + Schema | 秒级/API费用 | 有抖动需降噪 | 代码风格、推理合理性、回答质量、幻觉检测 |
| **人工评分器** | 领域专家 | 标注指南 | 分钟-小时级/人力 | 取决于标注员 | 校准 LLM、诊断异常、高风险兜底 |

**人工只用在六个必要场景**（专家时间 60%+ 花在校准和诊断上）：

1. 校准 LLM 评委——抽样 100-200 条对齐，一致率 ≥ 85% 才算可用
2. 主观任务打分——同理心、论证严谨度
3. 诊断通过率异常——0% 或 100% 通常是"任务/评分器坏了"
4. 建立 Ground Truth——新套件前 20-50 个参考解
5. Trace 采样审查——每周抽样读轨迹找隐藏失败
6. 高风险兜底——医疗/金融/安全场景 100% 复核

#### 三类评委的典型分工

| 检查内容 | 评委 | 示例 |
|---|---|---|
| 工具调用是否正确 | 确定性 | `tool_call contains "mcp"` |
| 产物文件是否存在 | 确定性 | `file_exists: ["report.md"]` |
| 响应包含关键信息 | 确定性 | `response_contains: ["测试有效", "CPU瓶颈"]` |
| Token/耗时在阈值内 | 确定性 | `max_tool_calls: 10` |
| 推理过程是否合理 | Rubric | LLM 按评分标准判断 |
| 是否存在幻觉 | Rubric | LLM 核查事实 |
| 主观质量判断 | 人工 | 专家审阅 |

### 3. 确定测评维度

五大维度，按优先级落地。P0 先保证"做对了"和"靠谱"，再逐步覆盖其他。

| 大类 | 回答的问题 | 优先级 | 主要评委 |
|---|---|---|---|
| 功能正确性 | 做对了吗？ | **P0** | 确定性 |
| 鲁棒性与安全 | 关键时刻翻车吗？ | **P0** | 确定性 + 人工 |
| 过程质量 | 过程合理吗？ | P1 | Rubric + 人工 |
| 效率与成本 | 划算吗？ | P1 | 确定性 |
| 体验与对齐 | 用户愿意继续用吗？ | P2 | Rubric + 人工 |

每个维度的子项和典型指标详见 `references/evaluation-dimensions.md`。

### 4. 设计用例集

用例从四个场景层层递进，每个场景都包含正向和负向用例：

```
触发 → 核心逻辑 → 产物质量 → 异常容错
```

#### 场景一：触发条件

验证 Skill 是否在正确场景被触发/不被触发。

- **正向**：各种合法表述都能触发
- **负向**：相似但不相关的 prompt 不误触发（过度触发比不触发更难发现）

#### 场景二：核心逻辑（用例量最大，通常是其他三类之和的 2-3 倍）

三步设计法：

1. **画出分支流程图**——梳理 Skill 所有核心分支路径
2. **每条分支至少 1 个用例**——覆盖所有核心逻辑分支
3. **补充组合和边界**——分支间组合、阈值边缘、跨分支冲突

分支过多时，优先覆盖**高频分支**和**易出错分支**。

#### 场景三：产物质量

验证最终产物——响应内容、输出文件、报告等是否完整、准确、可用。

检查项示例：
- 产物文件存在且格式正确
- 关键结论完整
- 不编造不存在的指标（幻觉检测）
- 不泄露敏感信息（PII、API key）

#### 场景四：异常容错

验证在异常输入、边界条件、环境故障下能否合理处理：

- **异常输入**：不存在的 ID、非法格式、缺少必填参数
- **边界条件**：大数据量、空数据、极限值
- **环境故障**：工具调用失败、超时、返回异常

#### 用例模板

每个用例应包含：

```yaml
# 基本信息
id: "trigger-pos-01"
type: "positive"  # positive / negative
scenario: "trigger"  # trigger / core-logic / output-quality / error-handling
description: "标准表述触发分析"

# 输入
prompt: "分析用例134263"

# 预期行为
expected_behavior:
  trigger: true    # 触发场景
  tool_calls:      # 核心逻辑场景
    - name: "mcp"
      contains: "tperf-mcp"
  outputs:         # 产物质量场景
    - file_exists: "report.md"
    - response_contains: ["测试有效"]
  error_handling:  # 异常容错场景
    on_invalid_id: "告知不存在，不编造结果"

# 评分规则
scoring:
  critical: true   # 关键判定（不一致直接扣大分）
  max_deduction: 10
```

### 5. 设计评分规则

采用**负分制**：初始 100 分，每有一项不符合预期扣对应分数，扣至 0 分。达标线建议 **80 分**（可根据风险等级调整）。

#### 评分维度框架

| 评分维度 | 关注点 | 典型扣分 |
|---|---|---|
| **关键判定** | 核心结论是否正确 | -80（直接不通过）|
| **过程** | 步骤遵循性、工具调用序列 | 每步不符合 -10 |
| **产物** | 章节内容、数据准确性 | 每项差异 -5，设上限 |
| **效率** | 耗时、Token 是否超标 | 基准为基础，每超标 10% -1，设上限 |
| **稳定性** | N 次执行是否全部达标 | 任一不达标 → 用例 0 分 |

#### 稳定性评估（pass^k）

对同一任务执行 N 次（建议 N=5），容忍阈值取决于 Skill 类型：

| 类型 | 容忍阈值 | 说明 |
|---|---|---|
| 关键决策类（问题定位、故障诊断）| 0%（N/N 全通过）| 任何一次失败都可能导致误判 |
| 辅助分析类（性能分析、报告生成）| ≤ 10% | 偶发偏差可接受，持续观察 |
| 创意生成类（代码建议、文案撰写）| ≤ 40% | 输出多样性本身是特性 |

#### 综合评分公式

```
单次 Trial:  trial_score = max(0, 100 + Σ扣分)
达标判定:    trial_pass  = (trial_score ≥ 80)
用例总分:    case_score  = avg(所有 trial_score)  若全部 trial_pass
                         = 0                       若任一 trial_pass = false
```

### 6. 建立基线

基线是**单个用例执行 1 次后，经人工确认的预期过程和预期结果的快照**。它回答"这个用例，正确的执行应该长什么样？"

#### 基线内容

| 类别 | 内容 |
|---|---|
| **预期过程** | 思维链（CoT）、工具调用序列（名称+参数+返回值+顺序）、中间产物 |
| **预期结果** | 最终响应文本、输出文件、报告内容 |
| **效率参考** | 耗时、Token 消耗 |

#### 基线建立流程

```
触发执行 → 采集 Trace + 产物 → 人工审核 → 确认通过 → 保存为基线
                                                     ↓
                                              不通过 → 调整 Skill 后重新执行
```

**关键原则**：用例设计阶段只定义 prompt 和检查规则，不需要手写预期结果。先跑出来，再确认，确认后就是预期。

#### 基线在评分中的作用

| 对比维度 | 确定性判定 | Rubric 判定 |
|---|---|---|
| 过程对比 | 工具调用序列是否一致、步骤是否缺失 | 思维链合理性、步骤选择是否最优 |
| 结果对比 | 关键内容是否缺失、产物是否一致 | 结论准确性、表述完整性 |
| 效率对比 | 超出基线 >50% → 效率退化 | — |

#### 基线更新时机

Skill 逻辑变更、模型升级、外部依赖变更、用例本身修改时，都需要重新执行并确认新基线。

### 7. 执行测评

#### 执行模式

| 模式 | 说明 | 适用场景 |
|---|---|---|
| 快速分析 | 每用例跑 1 轮 | 日常开发调试、确定基线 |
| 全量测评 | 每用例跑 N 轮 × M 模型 | 版本验收、模型对比 |
| 按模型筛选 | 只跑指定模型 | 单模型验证 |
| 重新评分 | 对已有结果重新评分 | 评分逻辑调整后无需重新执行 |

#### 触发时机

| 场景 | 触发方式 |
|---|---|
| Skill 提示词变更 | CI 自动触发（PR 合入前） |
| 模型版本升级 | 手动触发 |
| 定期巡检 | 定时触发 |
| 工具/依赖更新 | CI 自动触发 |

#### 环境要求

- 每次测评在隔离环境中执行（clone 到临时目录、每次重置 git 状态）
- 测评产物（Trace、日志、报告）统一归档
- 使用真实数据而非 mock 数据（保证可复现）

#### 前置依赖：Trace 输出能力

过程评测的前提是**能拿到结构化的执行轨迹**。被测 Agent/Skill 需输出：
- 工具调用记录（名称、参数、返回值、时间戳）
- 思维链/决策依据
- 中间产物

理想格式为 JSONL，每行一个事件，便于解析和逐步对比。

### 8. 维护用例集

#### 能力测评 vs 回归测评

| 类型 | 核心问题 | 期望通过率 | 作用 | 维护 |
|---|---|---|---|---|
| **能力测评** | "能把什么做好？" | 起步 20-50%，逐步提升 | 目标山峰，指导优化 | 主动拓展、频繁迭代 |
| **回归测评** | "原有能力还在吗？" | 接近 100% | 防御漂移，保住阵地 | 只增不减，CI 每次运行 |

**生命周期转化**：能力测评通过率稳定 = 100% 后，用例"毕业"到回归套件，从此每次 CI 都跑。

#### Bad Case 归入流程

线上/线下的 Bad Case 是最有价值的测评素材：

1. 复现 → 记录完整 Trace
2. 分析根因 → 确定预期行为
3. 修复 Agent/Skill
4. 将 Bad Case 纳入能力测评集
5. 通过率稳定后毕业到回归测评集

## 关键原则

- **能用代码判断的绝不用模型**：优先确定性评分器
- **基线是跑出来的，不是写出来的**：先执行再确认
- **核心逻辑用例量最大**：覆盖所有分支，每条分支至少 1 个用例
- **负向用例不能省**：过度触发比不触发更难发现，只测正例是假安全感
- **稳定性是硬指标**：1 次通过不代表稳定，N 次中有 1 次不通过就是风险
- **Bad Case 是黄金**：每个线上 Bad Case 都是最有价值的回归用例
- **人工时间花在刀刃上**：60%+ 用于校准 LLM 评委和诊断异常
- **0% pass^N 先排查任务定义**：稳定失败通常说明用例/评分器有问题，不是 Agent 不行
