# IT 业务系统技能包蒸馏

把面向人操作的 IT 业务系统逆向蒸馏为面向 AI 的独立 Skills 技能包。核心产物是 `distilled/` 目录：语义层让 AI 建立业务世界观，能力层让 AI 按业务用例选择 API，明细层让 AI 按需检索字段、枚举、错误码和冲突。

## 核心目标

交付可运行的领域能力说明书，不交付泛泛的系统介绍。

蒸馏后的技能包必须让 AI 能够：

- 理解业务对象、关系、边界、生命周期和关键术语；
- 从自然语言意图选择业务用例级 API，而不是自行拼装底层 CRUD；
- 区分服务端强制规则、服务端维护字段、调用方预判规则和副作用；
- 按需检索字段级 Schema、枚举、错误码、模型与代码冲突；
- 基于真实场景验证读、写、统计和规则反例是否可用；
- 只通过 API 操作业务系统，不直接访问数据库。

## 执行流程

1. 明确任务合同：
   - 目标系统名称和仓库/项目根目录；
   - 可用资料：原始需求、数据库 Schema、实例/种子数据、本体/领域模型、API 文档、服务层源码、路由/控制器源码；
   - 输出目录，默认 `<目标系统根目录>/distilled`；
   - 是否允许连接测试环境做真实 API 验证。
2. 先做资料盘点，再写文档。必须填写事实优先级表，并标注缺失资料。冲突裁决规则：
   - 实际行为/规则细节：源码 > 数据库约束 > 本体/领域模型 > 需求文档；
   - 业务意图/概念含义：需求文档 > 本体/领域模型 > 代码注释。
3. 创建输出骨架。优先运行：
   - `python3 <skill-dir>/scripts/init_distilled_package.py <output-dir> --system-name "<系统名>"`
   该脚本只生成标准目录和占位文件。必须在读完真实资料后替换占位内容；不得把占位文件当成完成品交付。
4. 建立语义层：
   - `README.md`：阅读顺序、加载策略、目录导航、资料盘点、写权限提醒；
   - `00-overview.md`：系统定位、业务边界、核心流程、AI 操作约定；
   - `01-glossary.md`：业务术语、别名、用户口语与代码名对齐；
   - `domain/`：一个核心业务对象一个文件，并维护 `domain/_index.md`。
5. 抽取业务规则并标注执行方。每条规则必须使用以下标签：
   - `【服务端强制】`：违反后会被接口/服务端拒绝；
   - `【服务端维护】`：由系统自动计算、更新或裁决；
   - `【调用方需预判】`：服务端不一定拦截，AI 调用前必须检查或向用户确认；
   - `【副作用】`：触发事件、重算、状态变更或其他连锁行为。
6. 建立能力层：
   - 以业务用例为粒度，一个文件描述一个完整业务动作；
   - 不把底层 CRUD 直接暴露为主要业务能力，除非 CRUD 本身就是完整业务动作；
   - 在 `api/_index.md` 写“自然语言意图 -> API 序列”的典型编排；
   - 剥离 UI 态字段，明确名称转 ID、主数据查询、前置校验和回显确认。
7. 建立明细层：
   - `reference/schema/*.md`：字段级 Schema，并映射到领域属性/API 字段；
   - `reference/enums.md`：真实枚举/字典值，区分代码已实现与模型/需求定义但未验证；
   - `reference/error-codes.md`：错误码、错误消息、来源和 AI 应对方式；
   - `reference/model-code-conflicts.md`：需求/本体/代码之间的所有差异。
8. 建立交叉引用。使用 `[[slug]]` 把对象、接口、Schema、枚举、错误码和冲突串起来，保证从“用户想做什么”能追溯到“该调什么 API、依赖什么对象、字段含义是什么”。
9. 验证：
   - 静态检查按 `references/validation.md` 执行；
   - 若允许动态验证，让 AI 只依赖 `distilled/` 推导 API 调用，再对测试环境实打接口，核对返回字段、状态流转、副作用和错误消息。
   - 最终必须运行 `python3 <skill-dir>/scripts/validate_distilled_package.py <output-dir>`。失败时继续补齐文档，不要交付失败状态。

## 资料处理规则

写最终结论前必须读取真实资料。大型代码库优先看路由/控制器，再看服务层/领域层，再看 Schema 和种子数据。用 `rg` 搜索业务动作、接口注册、校验器、状态流转、枚举定义、错误消息、重算逻辑和副作用。

发现资料冲突时，不要悄悄选一个版本。把实际运行行为写入对应对象/API 文档，同时把差异登记到 `reference/model-code-conflicts.md`。

资料不全时按以下方式降级：

- 只有代码：从代码反推对象、API 和规则；业务意图标注“由代码反推，待业务确认”；
- 只有需求：按需求定义“应有能力/API”，并标注“未实现/未验证”；
- 没有本体/领域模型：从 Schema 和服务层归纳对象关系，跳过本体-代码对比，但仍记录需求-代码冲突；
- 没有可用测试环境：只做静态验证，并在 `validation/results.md` 标注动态验证未执行原因。

## 输出结构

读取 `references/package-structure.md` 获取标准目录、分层职责、命名规则和加载策略。

读取 `references/templates.md` 获取各类文件模板。复制模板后必须用真实资料替换占位符，不要保留 `<待根据资料填写>`。

读取 `references/validation.md` 获取静态检查清单、动态场景设计和通过标准。

使用 `scripts/validate_distilled_package.py` 做交付前机器校验。该脚本检查标准文件、占位符、领域对象、用例级 API、Schema、规则标签、交叉引用和验证结论。它不能替代真实业务判断，但必须作为最终交付门槛。

维护或验收本技能自身时，可运行 `python3 <skill-dir>/scripts/smoke_agent_run.py --skill-dir <installed-skill-dir>`，验证项目启用路径能生成并校验一套最小标准 `distilled/` 包。

## 完成标准

只有同时满足以下条件，才算完成：

- 语义层短、准、可整体加载，没有字段级堆料；
- 每个核心业务对象都有一句话定义、状态/生命周期、关系、语义级属性、核心规则、相关能力和明细索引；
- 每个用例级 API 都有业务场景、能力边界、输入、输出、关键规则、涉及对象、可能错误、调用示例和写权限分级；
- 每条业务规则都有执行方标签；
- API 层出现的枚举、错误码和字段细节能在明细层查到；
- 需求/本体/代码冲突已记录；
- 有静态验证记录；若跳过动态验证，说明原因和剩余风险；
- `validate_distilled_package.py` 通过。

最终回复只报告输出路径、关键文件、验证结果和未验证事项。不要把整套 `distilled/` 文档粘贴到聊天里，除非用户明确要求。
