# 验证协议

同时验证文档完整性和运行可用性。静态检查只能说明结构基本合格；动态验证才能说明 AI 是否真的能靠 `distilled/` 操作业务系统。

## 静态检查清单

- [ ] 已填写资料盘点表，并标注缺失资料。
- [ ] 已写明事实优先级和冲突裁决规则。
- [ ] `00-overview.md` 足够短，且包含 AI 操作约定。
- [ ] 每个核心业务对象都有一句话定义、生命周期/状态、关系、语义级属性、核心规则、相关能力和明细索引。
- [ ] 每条业务规则至少有一个执行方标签：`【服务端强制】`、`【服务端维护】`、`【调用方需预判】`、`【副作用】`。
- [ ] `api/_index.md` 能把自然语言意图映射到 API 序列。
- [ ] 每个用例级 API 都有业务场景、能力边界、输入、输出、规则、对象、错误、示例和写权限分级。
- [ ] API 层是业务用例粒度；原子 CRUD 没有被当成主要业务能力，除非该 CRUD 本身就是完整业务动作。
- [ ] Schema 文件能把数据库/资源字段映射回领域属性和 API 字段。
- [ ] API 文档里出现的枚举和错误码都能在 `reference/` 查到。
- [ ] 冲突已登记到 `reference/model-code-conflicts.md`。
- [ ] 关键交叉引用使用 `[[slug]]`，且对象、接口、Schema、枚举和冲突之间能互相追溯。
- [ ] 已运行 `scripts/validate_distilled_package.py <distilled-dir>`，并修复所有失败项。

## 动态验证场景

在 `validation/scenarios.md` 准备场景，至少覆盖四类：

| 类型 | 验证目的 | 示例形态 |
| --- | --- | --- |
| 读/查询 | 名称转 ID、查询参数、筛选口径 | “查 X 部门今年的 Y” |
| 写/录入/更新 | 必填项、编排、服务端维护字段、确认流程 | “给客户 A 录入一笔……” |
| 统计/报表 | 指标含义、聚合口径、时间范围 | “未来三个月各部门预计收入多少” |
| 规则反例 | 错误预测、预判规则、错误解释 | “故意让比例不等于 100%” |

## 动态验证方法

1. 尽量使用新上下文或独立 agent。
2. 只提供 `distilled/` 技能包和用户场景，不额外透露答案。
3. 要求 agent 推导：
   - 应调用的 API 序列；
   - 每一步参数；
   - 调用前预判；
   - 预期成功/失败；
   - 写操作的回显确认话术。
4. 若允许访问测试环境，真实执行 API 调用。
5. 对比真实返回字段、状态流转、副作用、错误消息与文档是否一致。
6. 把证据、结论和缺口写入 `validation/results.md`。

## 通过标准

典型读、写、统计、规则反例场景都满足以下条件，才算动态验证通过：

- API 选择正确；
- 名称转 ID 或主数据查询正确；
- 参数结构和枚举值正确；
- 前置条件判断正确；
- 服务端维护字段和副作用处理正确；
- 非法场景能预测或解释真实错误；
- 全程不直接访问数据库。

如果跳过动态验证，必须记录原因，并把结论标注为“仅通过静态验证”。
